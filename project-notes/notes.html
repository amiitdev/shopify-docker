Part 1: Frontend Dockerfile

Multi-stage build with Node.js + Nginx

Complete Dockerfile

Dockerfile 

# Build stage
FROM node:20-alpine as build

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

ARG VITE_API_URL
ENV VITE_API_URL=$VITE_API_URL

RUN npm run build

# Production stage
FROM nginx:alpine

COPY --from=build /app/dist /usr/share/nginx/html

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
1

Build Stage

FROM node:20-alpine as build

Base image with Node.js 20 on Alpine Linux. as build names this stage for later reference.

WORKDIR /app

Sets working directory inside container. All subsequent commands run in /app.

COPY package*.json ./

Copies only dependency files first. This enables Docker layer caching for faster builds when dependencies don't change.

RUN npm install

Installs all dependencies defined in package.json. Creates node_modules inside container.

COPY . .

Copies all remaining source code into container. Now container has complete application.

ARG & ENV VITE_API_URL

ARG is build-time variable. ENV makes it available in container. Used by Vite to set API endpoint.

RUN npm run build

Executes vite build to create production-ready static files in /app/dist.

2

Production Stage

FROM nginx:alpine

Starts fresh with lightweight Nginx image. No Node.js included - smaller and more secure.

COPY --from=build /app/dist /usr/share/nginx/html

Copies built files from build stage to Nginx's default public directory. This is the multi-stage magic.

EXPOSE 80

Documents that container listens on port 80. Doesn't publish it - that happens with -p flag or in docker-compose.

CMD ["nginx", "-g", "daemon off;"]

Runs Nginx in foreground. Crucial for Docker - containers exit when main process stops. daemon off prevents Nginx from running in background.

Architecture Flow

Build Stage

Node.js Factory

Production Stage

Nginx Showroom

Factory builds product → Showroom displays finished product

Interview Question: Why multi-stage?

Q: Why not use Node in production?

A: Node is only needed to build. After build, only static files remain. Nginx is faster and lighter for static serving, resulting in smaller, more secure images.

Part 2: Backend Dockerfile

Node.js API server with Alpine Linux

Complete Dockerfile

Dockerfile (Backend) 

FROM node:20-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000

CMD ["node", "api/index.js"]

Line-by-Line Analysis

FROM node:20-alpine

Base image with Node.js 20 on Alpine Linux. Alpine is lightweight (~5MB) with smaller attack surface.

WORKDIR /app

Creates and sets /app as working directory. Equivalent to mkdir /app && cd /app.

COPY package*.json ./

Copies dependency files first for Docker layer caching. If source code changes but dependencies don't, Docker reuses npm install layer.

RUN npm install

Installs dependencies from package.json. Creates node_modules inside container.

COPY . .

Copies all remaining source code (API routes, controllers, etc.) into container.

Networking & Production

EXPOSE 3000

Documents that app listens on port 3000. To actually publish: docker run -p 3000:3000 myapp.

Left: host machine, Right: container

CMD ["node", "api/index.js"]

Default command when container starts. Server must be at /app/api/index.js.

Why CMD not RUN? RUN executes during build, CMD at runtime.

⚠️ Production Warning

This is OK for dev, but real production needs:

  • npm ci --only=production (faster, predictable)
  • Non-root user for security
  • Process manager (PM2)
  • Reverse proxy (Nginx)
  • Health checks
  • Proper logging

Dockerfile (Production-ready) 

FROM node:20-alpine

WORKDIR /app

COPY package*.json ./

# Use npm ci for production (faster, more reliable)
RUN npm ci --only=production

# Create non-root user for security

    adduser -S nodejs -u 1001

EXPOSE 3000

Part 3: Docker Compose

Multi-service orchestration for full-stack apps

Complete docker-compose.yml

docker-compose.yml 

version: '3.8'

services:
  backend:
    build: ./server
    ports:
      - '3000:3000'
    env_file:
      - ./server/.env

  frontend:
    build:
      context: ./client
      args:
        VITE_API_URL: http://localhost:3000
    ports:
      - '80:80'
    depends_on:
      - backend

Compose File Structure

version: '3.8'

Defines Docker Compose file format version. 3.8 is stable with good feature support.

services:

Root key where you define containers. Each service = one container. Compose orchestrates them as one application.

backend: section

Defines the backend service (Node.js API).

  • build: ./server - Builds from Dockerfile in ./server
  • ports: '3000:3000' - Maps host:container ports
  • env_file: ./server/.env - Loads environment variables
What Docker Compose Creates Automatically
Docker Network
Service DNS
Port Mapping
Startup Order

When you run docker compose up, Docker handles all of this automatically!

Frontend Service & Networking

frontend: section

Defines the frontend service (Vite + Nginx).

  • context: ./client - Build context folder
  • args: VITE_API_URL - Build argument for frontend
  • ports: '80:80' - Serves on standard HTTP port
  • depends_on: - backend - Startup dependency
⚠️ Critical Networking Issue

VITE_API_URL: http://localhost:3000 is WRONG for Docker!

Inside container, localhost means the container itself, not the host machine.

Correct approach: Use service name http://backend:3000

depends_on: nuance

depends_on only ensures startup order, not readiness.

For production: add healthchecks to ensure backend is ready before frontend tries to connect.

Production Improvements

Internal Service Name

Use VITE_API_URL=http://backend:3000 for container-to-container communication.

Restart Policy

Add restart: always to automatically restart containers on failure.

Health Checks

Add healthcheck configurations to ensure services are actually ready, not just started.

Interview Question: Container Communication

Q: How does frontend container communicate with backend?

A: Through Docker's internal network using service name (backend) as hostname. Docker Compose automatically creates a network and DNS so services can find each other by name.

Common mistake: Using localhost inside container (points to container itself, not other services).
