on:
  push:
    branches:
      - master

jobs:
  build-and-push:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and Push Backend
        uses: docker/build-push-action@v5
        with:
          context: ./server
          push: true
          tags: amiitdev/shopify-backend:latest

      - name: Build and Push Frontend
        uses: docker/build-push-action@v5
        with:
          context: ./client
          push: true
          tags: amiitdev/shopify-frontend:latest

Understanding GitHub Actions Workflow

This is a GitHub Actions Workflow File

Written in YAML

YAML is a human-readable data format. Indentation matters!

Location

Stored in: .github/workflows/

Automatically Executed

GitHub automatically runs it when conditions are met.

Runs on GitHub's Servers

Uses GitHub's cloud infrastructure, not your computer.

What It Tells GitHub

"When X happens, do Y."

X

Trigger Condition

When code is pushed to master branch

Y

Actions to Perform

Build Docker images and push to Docker Hub

Lines 2-5: Trigger Conditions 

on:

This is VERY Important

This section defines WHEN the workflow runs.

Breaking It Down

on: "When should this workflow run?"

This is the trigger keyword. Everything indented under it defines the conditions.

push: "Run when someone pushes code"

Key terms:

  • Push = git push
  • Not commit, not pull request, not merge
  • Only when code is pushed to remote repository
branches: "Which branches trigger this?"

Limits where the workflow runs.

- master "Only run for master branch"

The dash (-) means it's a list item.

Real World Example with 5 Developers

DOES NOT Run

  • Developer pushes to feature/login
  • Developer pushes to bugfix/header
  • Developer pushes to dev

DOES Run

  • Developer merges to master
  • Direct push to master

This protects production. Only changes merged to master trigger deployment.

Line 11: steps:
What are steps?

A job contains steps. Steps are executed one by one, in order.

Important Rule

If one step fails ‚Üí the job stops ‚Üí no more steps execute.

This is called "fail fast" - prevents broken code from progressing.

Step 1: Checkout Repository 

  - name: Checkout repository
    uses: actions/checkout@v4
Critical Understanding

When GitHub starts a new Ubuntu machine:

  • It is EMPTY
  • It does NOT have your project code
  • This step downloads your repository

Without this step, nothing else would work!

Step Details

uses:

Means: "Use a prebuilt GitHub Action"

GitHub Actions has a marketplace of reusable actions.

actions/checkout@v4
Official GitHub Action

This is GitHub's official checkout action.

Version 4 (latest stable).

What It Actually Does:
git clone https://github.com/your-username/your-repo.git

Inside the runner machine, it clones your repository.

Step 2: Login to Docker Hub 

  - name: Log in to Docker Hub
    uses: docker/login-action@v3
    with:
      username: ${{ secrets.DOCKER_USERNAME }}
      password: ${{ secrets.DOCKER_PASSWORD }}
Why Login?

Because later we will push Docker images to Docker Hub.

You cannot push images unless you're logged in (authenticated).

Understanding Each Part

uses: docker/login-action@v3
Official Docker Action

Docker's official GitHub Action for logging in.

with:

Means: "Pass parameters to the action"

Think of it as function arguments.

${{ secrets.DOCKER_USERNAME }}

This pulls a secret stored securely in GitHub.

It is NOT visible in logs. It is encrypted.

Why not hardcode?

  • Hardcoding leaks credentials
  • Anyone can see it in the repository
  • It's insecure
  • Secrets protect credentials
GitHub Secrets

Encrypted

Secure Storage

No Logs

Access Control

Stored in: GitHub ‚Üí Settings ‚Üí Secrets and variables ‚Üí Actions
