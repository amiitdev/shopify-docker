Line-by-Line Deep Dive

1
apiVersion: apps/v1

This tells Kubernetes we're using the apps API group, version v1. Deployments belong to this group.

ü§î Why? Different Kubernetes objects live in different API groups. Services use apiVersion: v1 (core group), Deployments use apps/v1.

2
kind: Deployment

We are creating a Deployment object.

ü§î Why Deployment and not Pod? Deployment gives you:

  • Auto-restart if pod crashes
  • Scaling (replicas)
  • Rolling updates (zero downtime)
  • Self-healing
3-4
metadata: name: backend-deployment

The name of this Deployment object in Kubernetes.

ü§î Why name matters: You use this name to control the deployment:

kubectl rollout restart deployment backend-deployment
kubectl scale deployment backend-deployment --replicas=3
kubectl describe deployment backend-deployment
6
replicas: 2

We want 2 identical pods running.

ü§î Why 2 replicas?

High Availability

If one crashes, other still serves

Load Distribution

Traffic shared between 2 pods

7-9
selector: matchLabels: app: backend

Deployment selects Pods with label app: backend.

ü§î CRITICAL CONCEPT: Deployment must know which Pods it manages. This selector MUST match the Pod template labels (lines 12-13).

‚ö†Ô∏è If mismatch: Deployment creates pods but can't manage them!

12-13
labels: app: backend

Every Pod created gets this label.

ü§î Why labels matter:

  • Deployment selector matches this (management)
  • Service selector matches this (networking)
  • You can query pods by label: kubectl get pods -l app=backend

name: backend

Container name inside the Pod.

ü§î Why name containers? Useful for:

Multi-container pods

Logs: kubectl logs pod-name -c backend

ü§î What's in this image?

  • Compiled from your source code
  • Published to Docker Hub by CI/CD
  • Tagged as :latest (in production, use specific versions!)

ü§î Why Secrets not environment variables?

ü§î CRITICAL FOR PRODUCTION:

Maximum allowed

‚úÖ Why limits matter:

  • Prevents one pod from eating all memory
  • Prevents node crash (OOM Killer)
  • Allows proper scheduling
  • Required for auto-scaling (HPA)

Checks if pod is ready to receive traffic.

ü§î What it does:

200 OK

‚úÖ Consequence if fails: Pod removed from Service load balancer. No traffic sent until healthy.

‚úÖ Consequence if fails: Container restarted automatically (self-healing).

Line-by-Line Deep Dive

1
apiVersion: v1

We're using the core Kubernetes API group (v1).

ü§î Why v1 and not apps/v1? Services belong to the core API group, not the apps group. Different resource types live in different API groups.

2
kind: Service

We are creating a Service object.

ü§î Why Service? Service gives you:

  • Stable internal IP address
  • DNS name inside the cluster
  • Load balancing across pods
  • Abstracts away pod IP changes
3-4
metadata: name: backend

The name of this Service object.

ü§î Why this name matters: This becomes your internal DNS name!

Inside the cluster, frontend can call:

http://backend:3000

Kubernetes automatically creates DNS entry:

backend.default.svc.cluster.local
6-7
selector: app: backend

This Service selects pods with label app: backend.

ü§î CRITICAL: This must match your Deployment Pod labels!

Remember in Deployment:

labels:
  app: backend

‚ö†Ô∏è If selector doesn't match labels:

  • Service exists but routes to zero pods
  • App breaks silently
  • Common Kubernetes bug
8-10
ports: port: 3000 targetPort: 3000

Defines port mapping between Service and Pods.

ü§î Key difference:

port: 3000

Service port (internal cluster)

Frontend calls: backend:3000

targetPort: 3000

Container port (where app listens)

Container listens: 0.0.0.0:3000

You could also do:

port: 80
targetPort: 3000

Then frontend would call backend:80 but traffic would go to container port 3000.

11
type: ClusterIP

This Service is accessible only inside the cluster.

ü§î Why ClusterIP for backend?

  • Backend should NOT be exposed to public internet
  • Only frontend needs to access backend
  • Security best practice
  • Clean internal networking

üìö Service Types:

ClusterIP

Internal only (default)

NodePort

Exposed on node port

LoadBalancer

Cloud external LB

ExternalName

Maps to external DNS

Big Picture: What Happens Internally

‚úÖ What This Service Creates:

  • Stable virtual IP (10.x.x.x range)
  • DNS name: backend.default.svc.cluster.local
  • Load balancing across backend pods
  • Automatic pod discovery
  • Abstracts pod IP changes

üåê Network Flow:

Frontend

Service

Backend Pods

Frontend ‚Üí Service (Load Balancer) ‚Üí Backend Pods

üî• Important Debug Command:

kubectl get endpoints backend

This shows actual pod IPs that the Service is connected to.

ü§î What is Ingress?

  • Traffic router inside Kubernetes
  • HTTP/HTTPS reverse proxy
  • Domain-based routing
  • Path-based routing
  • Central entry point
3-4

Name of the Ingress object.

ü§î For management:

7

This rule applies only when user visits shopify.local.

ü§î Domain-based routing:

api.shopify.local ‚Üí backend (could add)
admin.shopify.local ‚Üí admin (could add)

You need to add to /etc/hosts:

10-11

ü§î Prefix pathType means:

  • / ‚Üí matches
  • /api ‚Üí matches
  • /shop ‚Üí matches
  • /admin ‚Üí matches

All traffic goes to frontend, then nginx proxies /api to backend.
