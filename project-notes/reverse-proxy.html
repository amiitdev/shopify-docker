Key Concept: The Gatekeeper Pattern

Nginx becomes the single entry point (gatekeeper) for all traffic. It decides:

  • / → Serve frontend static files
  • /api/* → Forward to backend (internally)
  • /assets/* → Serve static assets

Browser only sees http://localhost. Backend becomes invisible.

Step 2: Create Nginx Configuration

Create client/nginx.conf in your frontend project:

client/nginx.conf 
server {
    listen 80;

    # Serve frontend files for root path
    location / {
        root   /usr/share/nginx/html;
        index  index.html;
        try_files $uri /index.html;
    }

    # Forward API requests to backend
    location /api/ {
        proxy_pass http://backend:3000/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

Nginx Directives Explained

listen 80;

Nginx listens on port 80 inside container

location / { ... }

Handles all requests to root path. Serves your React/Vue/Angular app.

try_files $uri /index.html;

Crucial for SPAs! If file doesn't exist, serve index.html. This enables client-side routing.

Reverse Proxy Magic

location /api/ { ... }

Catches all requests starting with /api

proxy_pass http://backend:3000/;

MAGIC LINE! Forwards requests to backend container. backend is service name in Docker Compose.

proxy_set_header directives

Passes client IP and headers to backend. Backend knows original requester.

Interview Question: SPA Routing

Q: Why do we need try_files $uri /index.html; in nginx config?

A: Single Page Applications (React, Vue, Angular) use client-side routing. When user visits /dashboard, there's no actual dashboard.html file. Nginx needs to serve index.html anyway, then JavaScript handles the route.

Without this, refreshing on any route would give 404 error.

3

Step 3: Update Frontend Dockerfile

Complete Dockerfile 

# Build stage
FROM node:20-alpine as build

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

ARG VITE_API_URL
ENV VITE_API_URL=$VITE_API_URL

RUN npm run build

# Production stage
FROM nginx:alpine

COPY nginx.conf /etc/nginx/conf.d/default.conf
COPY --from=build /app/dist /usr/share/nginx/html

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]

Key Changes Explained

COPY nginx.conf /etc/nginx/conf.d/default.conf

NEW LINE! Copies custom Nginx config to replace default. This enables reverse proxy functionality.

COPY --from=build /app/dist /usr/share/nginx/html

Copies built frontend files from build stage. No source code, no node_modules - just static files.

Multi-Stage Build Benefits
  • Build stage has Node.js (for building)
  • Production stage has only Nginx + static files
  • Smaller final image (~20MB vs ~350MB)
  • More secure (less attack surface)

Step 4: Update Frontend API URL

Change Docker Compose Build Args

docker-compose.yml (frontend section) 
frontend:
  build:
    context: ./client
    args:
      VITE_API_URL: /api   ← CHANGED FROM http://localhost:3000
  ports:
    - '80:80'
  depends_on:
    - backend
The Beauty of This Change

Frontend now calls /api (relative URL). Nginx handles routing. Clean separation!

How Frontend Code Works Now

React/Vue/Axios Example 
// Before (connecting directly to backend)
axios.get('http://localhost:3000/products')

// After (using environment variable)
axios.get(`${import.meta.env.VITE_API_URL}/products`)

// When built with VITE_API_URL=/api, becomes:
axios.get('/api/products')

// Browser makes request to:
http://localhost/api/products

// Nginx intercepts and forwards to:
http://backend:3000/products (internal Docker network)
URL Transformation Flow

Frontend → /api/products → Nginx → backend:3000/products → Backend

Final Production Configuration

Complete, production-ready docker-compose.yml

docker-compose.yml (Production Version) 
version: '3.8'

services:
  # Backend service - NO PORT EXPOSED
  backend:
    build: ./server
    env_file:
      - ./server/.env
    restart: unless-stopped

  # Frontend service - single entry point
  frontend:
    build:
      context: ./client
      args:
        VITE_API_URL: /api
    ports:
      - '80:80'
    depends_on:
      - backend
    restart: unless-stopped
Single Entry Point

Only port 80 exposed. Clean, professional architecture.

Internal Networking

Backend accessible only within Docker network via service name.

Auto-Restart

restart: unless-stopped ensures services recover from crashes.

⚠️ Important: Rebuild Required

Because Vite injects environment variables at build time, you must rebuild with no cache:

docker compose down
docker compose build --no-cache
docker compose up

Without --no-cache, Docker might use cached layers with old VITE_API_URL value.
